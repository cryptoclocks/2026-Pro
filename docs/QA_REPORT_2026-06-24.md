## 8. ความเสี่ยง / ข้อสังเกตที่ควรรู้ก่อนเทสต่อ

1. **TC-DV-12 spec mismatch** ⚠️ — prompt เขียนว่า `DELETE /file` แต่เฟิร์มแวร์ใช้ **`POST /delete`**.
   - **ทางเลือก A**: แก้ prompt (`docs/QA_TEST_PROMPT_MINIMAX.md`) ให้ตรงกับโค้ด
   - **ทางเลือก B**: เพิ่ม `DELETE /api/v1/file` route ใน `local_api.c` ให้ตรงกับ REST convention
   - **ทางเลือก C**: เปลี่ยน user web/app ให้เรียก `POST /delete` แทน

2. **`POST /api/v1/payloads/validate` ตอบ 400 ตอน body ว่าง** — ดูเหมือน body validation รันก่อน auth (หรือ endpoint เปิดให้ public เรียกได้). ถ้า intended → โอเค; ถ้า intended ให้มี JWT → ต้องแก้ guard

3. **User web เป็น hash-routed SPA ไม่ใช่ path-routed** — prompt ระบุเป็น "หน้า/แท็บ Connect · System · Profile …" ตามที่อ่านจาก `index.html` จริง ๆ แล้วทุกอย่างอยู่ในหน้าเดียวและสลับ view ด้วย JS state ผู้เทสต่อต้อง navigate ผ่าน in-app links เท่านั้น

4. **Local API บนเครื่อง "Unauthenticated on the local network in v1"** — อ่านจาก `local_api.h:5` — ถือว่าเครื่องอยู่ใน LAN ที่เชื่อถือได้เท่านั้น ถ้าเทสในเครือข่ายที่ไม่น่าเชื่อถือ → ควร disable local API

### 14.2 ข้อค้นพบเกี่ยวกับ /files

**Spec mismatch ที่ prompt ไม่ได้บอก**:
- Endpoint ใช้ query key **`dir`** ไม่ใช่ `path` (ดู `local_api.c:251`)

- Path ต้องอยู่ใต้ `pages/` เท่านั้น (no `/sd/config`, no `/sd/PACKAGES`, etc.)

```c
// local_api.c:177-180
static bool rel_path_safe(const char *p) {
    return p && strncmp(p, "pages/", 6) == 0 && !strstr(p, "..") && !strchr(p, '\\');
}
```

ผู้เทส user-web ต้องอัพเดท: ถ้า Flutter app เรียก `?path=...` จะพัง — ต้องเรียก `?dir=...` และ path ต้อง lowercase

### TC-LIVE-10: brightness out-of-range validation (edge case)

**ลำดับ**:
1. POST /brightness `{"value":999}` → 200 `{"ok":true"}`
2. POST /brightness `{"value":-50}` → 200 `{"ok":true"}`
3. GET /info → brightness=0 (clamped)

**ผล**: ⚠️ **MINOR BUG** — error message "need {value:0-100}" โผล่เฉพาะตอน value ไม่ใช่ตัวเลข (string)
ตอน value เป็นตัวเลขนอกช่วง → **firmware ตอบ ok=true แล้ว clamp เงียบ ๆ**
- `value=999` → clamped to 100
- `value=-50` → clamped to 0

**ตำแหน่ง**: `firmware/components/local_api/local_api.c:147-164`
```c
if (!cJSON_IsNumber(v)) { ... return 400 "need ..." }   // only checks type, not range
ccp_board_set_brightness(v->valueint);                    // setter clamps silently
```

**ความเสี่ยง**: ต่ำ — clamp ทำงานถูก แต่ error message misleading
**ทางแก้**: เพิ่ม range check ใน handler → return 400 ถ้า value < 0 || > 100

**สำคัญมาก**: หลัง TC-LIVE-12 (bad config POST) → LVGL hang ก็เกิดอีกครั้ง หลังเวลาผ่านไป ~3 นาที

**ผล**: ❌ **CONFIRMED — LVGL hang เป็น bug แยก ไม่ใช่ side-effect ของ crypto crash**

**อาการร่วม**:
- Watchdog fires ทุก 10 วินาที
- HTTP API ค้าง (curl timeout exit 28)
- Serial console (`pages`, `ver`, `heap`) ยังตอบ

- LVGL task บน CPU 1 ไม่ reset watchdog → เป็นไปได้ว่า

  2. memory corruption จาก atomic write race (FAT ไม่มี atomic rename จริง)
  3. WAMR widget render loop ไม่มี watchdog feeding
- Backtrace มี addresses 0x4203xxxx-0x4205xxxx (LVGL widget code)

**ทางแก้ (ต้องทำ P0 ก่อน ship)**:
1. **A**: เพิ่ม watchdog reset ใน LVGL task (ถ้ายังไม่มี)
2. **B**: ลด WAMR widget complexity (refactor heavy widgets)
3. **C**: เพิ่ม timeout ใน `display_engine_lock()` และ log เมื่อ timeout > 1s
4. **D**: ใช้ LVGL timer-based render loop ที่ feed watchdog ทุก iteration
5. **E**: ตรวจ memory integrity หลัง atomic write (FAT unlink+rename race condition)
