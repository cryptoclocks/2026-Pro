# Supabase Device Configuration Schema

Status: proposed design — review before migration/implementation  
Date: 2026-06-23

## 1. Core decisions

1. Supabase/Postgres is the source of truth for display configuration.
2. Settings belong to a **device**, not directly to a user email.
3. A user owns one or more devices through `Device.ownerId`; each device has its
   own page settings.
4. Web User, Mobile, and Admin read/write settings through the OCI API. They do
   not read configuration from MQTT and do not write Supabase tables directly.
5. MQTT carries small change notifications and acknowledgements. Large config
   documents and files move over HTTPS.
6. The device keeps a last-known-good copy on SD/LittleFS so it remains usable
   while offline.
7. Images/GIFs are files in a private object-storage bucket. JSON stores only
   logical asset references and metadata, never base64 data or expiring signed
   URLs.

The external hardware id remains `Device.deviceId`, for example
`ccp-983daee91478`. Database relations use the internal `Device.id`.

## 2. Ownership model

```text
Supabase auth user
        │
        ▼
      User
        │ ownerId
        ▼
      Device
        ├── DeviceConfigHead       one current global revision
        ├── DevicePageSettings[]   one row per page
        ├── DeviceAsset[]          logical assets for that device/page
        ├── DeviceConfigRevision[] audit and rollback history
        └── DeviceSyncState        desired vs applied revision
```

Display profile data such as nickname, role, motto, or company is content shown
on a particular clock. It is not the account profile and must not be copied into
`User`.

Owner transfer does not silently rewrite display content. Admin must explicitly
choose either:

- keep the existing display settings; or
- reset the device to defaults for the new owner.

If a user wants the same Profile or Clock theme on several devices, add an
optional `UserConfigPreset` later. Applying a preset copies a validated snapshot
to selected devices and creates normal device revisions. Devices must not read
live settings directly from a user preset because that would make one edit
unexpectedly modify every owned clock.

Data that must remain outside page config:

- ownership and customer identity;
- entitlements and purchases;
- online status and telemetry;
- device/claim/MQTT tokens;
- audit logs;
- temporary signed URLs;
- social-network fetch/cache diagnostics.

## 3. Tables

### 3.1 Existing `User`

Add a stable Supabase Auth subject. Email must not be the authentication join
key because an email can change.

| Column | Type | Rules |
|---|---|---|
| `id` | text | Existing internal primary key |
| `authSubject` | uuid/string | Unique; value of Supabase JWT `sub` |
| `email` | text | Unique contact/login display value |
| `name` | text nullable | Account name, not clock profile name |
| `role` | enum | `USER`, `ADMIN`, `SUPER_ADMIN` |

### 3.2 Existing `Device`

Keep device identity, ownership, and telemetry here. During migration,
`settings` and `settingsVersion` remain as a compatibility cache only.

| Column | Type | Purpose |
|---|---|---|
| `id` | text | Internal primary key used by relations |
| `deviceId` | text | Unique external hardware id |
| `ownerId` | text nullable | Current owner |
| `serialNumber` | text nullable | Human-facing `CCP-000001` style serial |
| `mac` | text nullable | Immutable Wi-Fi station MAC |
| `online`, `lastSeenAt` | status fields | Current observed state |
| `settings` | jsonb | Deprecated compiled cache during migration |
| `settingsVersion` | integer | Deprecated mirror of global revision |

### 3.3 `DeviceConfigHead`

Exactly one row per device. It coordinates global revisioning and holds
device-wide values.

| Column | Type | Rules |
|---|---|---|
| `deviceDbId` | text | PK/FK → `Device.id`, cascade delete |
| `revision` | bigint | Monotonic desired configuration revision |
| `systemConfig` | jsonb | Brightness, display mode and global behavior |
| `compiledConfig` | jsonb | Derived firmware-compatible snapshot |
| `compiledSha256` | char(64) nullable | Hash of canonical compiled JSON |
| `updatedByUserId` | text nullable | FK → `User.id` |
| `updatedSource` | text | `web_user`, `mobile`, `admin`, `system`, `device_import` |
| `updatedAt` | timestamptz | Server timestamp |

`compiledConfig` is a cache generated by the API. It must never be edited
directly. Canonical values live in `systemConfig`, `DevicePageSettings`, and
asset tables.

Initial `systemConfig`:

```json
{
  "display_mode": "static",
  "page_delay_s": 10,
  "brightness": 80
}
```

Validation:

- `display_mode`: `static | dynamic`
- `page_delay_s`: integer `3..3600`
- `brightness`: integer `5..100`

### 3.4 `PageSettingSchema`

Immutable schema versions used by the API and all clients to validate and
render settings forms.

| Column | Type | Rules |
|---|---|---|
| `pageSlug` | text | Part of PK, e.g. `profile` |
| `schemaVersion` | integer | Part of PK; never mutate published versions |
| `packageId` | text nullable | e.g. `com.ccp.profile` |
| `jsonSchema` | jsonb | Machine validation contract |
| `uiSchema` | jsonb | Labels, groups, order and widget hints |
| `defaultConfig` | jsonb | Defaults for a new device |
| `assetSlots` | jsonb | Allowed file slots and transformations |
| `createdAt` | timestamptz | Server timestamp |

Native and Store pages use the same table. A published `PayloadVersion`
references one schema version rather than embedding a mutable schema.

### 3.5 `DevicePageSettings`

One current row for every page installed or configured on a device.

| Column | Type | Rules |
|---|---|---|
| `id` | text/uuid | Primary key |
| `deviceDbId` | text | FK → `Device.id`, cascade delete |
| `pageSlug` | text | `clock`, `crypto`, `profile`, etc. |
| `schemaVersion` | integer | Schema used to validate `config` |
| `pageRevision` | bigint | Increments only when this page changes |
| `enabled` | boolean | Included in the device rotation |
| `position` | smallint | Rotation order; unique per device when enabled |
| `config` | jsonb | Canonical page-specific values |
| `updatedByUserId` | text nullable | Last actor |
| `updatedSource` | text | Same source values as config head |
| `updatedAt` | timestamptz | Server timestamp |

Constraints and indexes:

```text
UNIQUE(deviceDbId, pageSlug)
INDEX(deviceDbId, enabled, position)
CHECK(position >= 0)
CHECK(pageRevision >= 0)
```

Do not keep `pages: ["clock", ...]` as canonical data. The API derives it from
`enabled=true ORDER BY position`.

### 3.6 `DeviceAsset`

A stable logical asset slot. Replacing an avatar creates a new immutable
version; it does not change the logical asset id.

| Column | Type | Rules |
|---|---|---|
| `id` | text/uuid | Primary key |
| `deviceDbId` | text | FK → `Device.id`, cascade delete |
| `pageSlug` | text | Owning page |
| `assetKey` | text | `avatar`, `background`, or generated photo key |
| `kind` | text | `image`, `gif`, `audio`, `font`, `binary` |
| `enabled` | boolean | Included in current manifest |
| `sortOrder` | integer nullable | Slideshow ordering |
| `currentVersionId` | text nullable | FK → current immutable file version |
| `createdAt`, `updatedAt` | timestamptz | Server timestamps |

Constraints:

```text
UNIQUE(deviceDbId, pageSlug, assetKey)
```

Built-in asset slots:

| Page | Asset key | Rules |
|---|---|---|
| Profile | `avatar` | PNG, 132×132, max 300 KB |
| Weather | `background` | GIF, max 480×320, size limit set by API |
| Slideshow | generated UUID key | PNG/JPEG, max 480×320 |

### 3.7 `DeviceAssetVersion`

Immutable metadata for one uploaded file.

| Column | Type | Rules |
|---|---|---|
| `id` | text/uuid | Primary key |
| `assetId` | text/uuid | FK → `DeviceAsset.id`, cascade delete |
| `version` | integer | Monotonic within logical asset |
| `bucket` | text | Normally `device-assets` |
| `objectPath` | text | Unique private object path |
| `contentType` | text | Validated MIME type |
| `sizeBytes` | bigint | Validated file size |
| `sha256` | char(64) | Device integrity check |
| `width`, `height` | integer nullable | Image dimensions |
| `createdByUserId` | text nullable | Upload actor |
| `createdAt` | timestamptz | Server timestamp |

Object path convention:

```text
devices/<external-device-id>/pages/<page-slug>/<asset-key>/v<version>.<ext>
```

Example:

```text
devices/ccp-983daee91478/pages/profile/avatar/v3.png
```

Signed URLs are generated on demand and are never stored in this table.

### 3.8 `DeviceConfigRevision`

Immutable audit history. One row is written in the same transaction as every
config or asset change.

| Column | Type | Purpose |
|---|---|---|
| `id` | text/uuid | Primary key |
| `deviceDbId` | text | Affected device |
| `globalRevision` | bigint | Resulting `DeviceConfigHead.revision` |
| `pageSlug` | text nullable | Null for a system-only change |
| `pageRevision` | bigint nullable | Resulting page revision |
| `changeType` | text | `settings`, `asset`, `enable`, `reorder`, `reset`, `import` |
| `source` | text | Web/mobile/admin/device/system |
| `actorUserId` | text nullable | User that caused the change |
| `beforeConfig` | jsonb nullable | Previous page/system document |
| `afterConfig` | jsonb nullable | New page/system document |
| `metadata` | jsonb | Asset id, client version, reason, etc. |
| `createdAt` | timestamptz | Server timestamp |

Keeping full page documents is acceptable because page settings are small and
it makes audit and rollback reliable.

### 3.9 `DeviceSyncState`

One row per device recording what the server wants and what the physical unit
has successfully applied.

| Column | Type | Purpose |
|---|---|---|
| `deviceDbId` | text | PK/FK → `Device.id` |
| `desiredRevision` | bigint | Normally equals config-head revision |
| `reportedRevision` | bigint | Last revision acknowledged by device |
| `reportedSha256` | char(64) nullable | Hash acknowledged by device |
| `status` | text | `pending`, `syncing`, `applied`, `failed`, `offline` |
| `lastNotifiedAt` | timestamptz nullable | Last MQTT notification |
| `lastAppliedAt` | timestamptz nullable | Successful device ACK |
| `lastError` | text nullable | Last device/download/apply error |
| `updatedAt` | timestamptz | Server timestamp |

The Admin page must display both:

```text
Online: true/false
Configuration: applied v42 / desired v42
```

Online status alone does not prove that settings were applied.

## 4. Canonical JSON contracts by page

The examples below are version 1 canonical documents. Unknown keys are rejected
unless a published page schema explicitly allows them.

### 4.1 Profile

```json
{
  "show": true,
  "name": "Natthapong Suwanjit",
  "nickname": "SAT",
  "role": "Developer",
  "motto": "DON'T TRUST  VERIFY",
  "company": "CryptoClock",
  "colors": {
    "name": "#F0B90B",
    "role": "#848E9C",
    "company": "#848E9C",
    "verify": "#00D1FF",
    "background": "#0B0E11"
  },
  "social": {
    "facebook": {
      "url": "",
      "followers": "",
      "following": "",
      "secondary_label": "Following"
    },
    "youtube": {
      "url": "",
      "followers": "",
      "following": "",
      "secondary_label": "Following"
    },
    "tiktok": {
      "url": "",
      "followers": "",
      "following": "",
      "secondary_label": "Following"
    },
    "instagram": {
      "url": "",
      "followers": "",
      "following": "",
      "secondary_label": "Following"
    }
  }
}
```

The avatar is the `DeviceAsset` with `pageSlug=profile`,
`assetKey=avatar`. It is not embedded in this JSON.

Limits:

- text fields: trim whitespace, max 80 characters;
- motto: max 120 characters;
- URLs: HTTPS only, max 500 characters;
- colors: uppercase `#RRGGBB`;
- follower values are display strings, max 24 characters.

The firmware compiler flattens this document to the current keys
`name_color`, `fb_url`, etc. and injects:

```json
{
  "avatar": "pages/profile/assets/avatar.png"
}
```

### 4.2 Clock

```json
{
  "format_24h": true,
  "show_seconds": true,
  "show_date": true,
  "show_logo": false,
  "date_format": "D MMM YYYY",
  "tz_offset_min": 420,
  "colors": {
    "time": "#00D1FF",
    "seconds": "#FF9500",
    "date": "#848E9C",
    "background": "#0B0E11"
  },
  "alarms": [
    {
      "id": "alarm-uuid",
      "time": "07:00",
      "label": "Wake up",
      "days": [1, 2, 3, 4, 5],
      "enabled": true,
      "sound": "beep"
    }
  ]
}
```

Limits:

- timezone: `-720..840` minutes;
- date format: one of the published options;
- at most 8 alarms;
- alarm time must match `HH:mm`;
- days contain unique integers `1..7`.

### 4.3 Crypto

```json
{
  "symbols": ["BTCUSDT", "ETHUSDT"],
  "style": "chart",
  "timeframe": "15m",
  "currency": "USD",
  "fetch_interval_s": 10,
  "alerts": [
    {
      "id": "alert-uuid",
      "symbol": "BTCUSDT",
      "direction": "above",
      "price": 100000,
      "enabled": true
    }
  ]
}
```

Limits:

- unique uppercase symbols, maximum defined by the published schema;
- `style`: `chart | big`;
- `timeframe`: `1m | 5m | 15m | 1h | 4h | 1d`;
- `currency`: initially `USD | THB`;
- refresh interval: `5..3600`;
- at most 8 alerts; price must be positive.

If cloud notifications are added later, alert execution state belongs in a
separate operational table. The display preference stays in this config.

### 4.4 Slideshow

```json
{
  "order": "upload",
  "effect": "fade",
  "interval_s": 6
}
```

Photos are `DeviceAsset` rows with `pageSlug=slideshow`. `sortOrder` controls
manual order.

Limits:

- `order`: `upload | name | random`;
- `effect`: `fade | slide | none`;
- interval: `2..3600`.

### 4.5 Weather

```json
{
  "location": {
    "label": "Bangkok",
    "latitude": 13.7563,
    "longitude": 100.5018,
    "timezone": "Asia/Bangkok"
  },
  "show_pm": true,
  "background_mode": "auto",
  "background_preset": null,
  "bg_color": "#0B0E11"
}
```

`background_mode`:

- `auto`: choose animation from weather condition;
- `preset`: use `clear`, `cloudy`, `rain`, `snow`, or `none`;
- `custom`: use the Weather `background` asset;
- `none`: no animation.

The API may accept legacy `{city, bg_gif}` input but must convert it to this
canonical shape.

### 4.6 Calendar

```json
{
  "title": "CALENDAR",
  "bg_color": "#0B0E11",
  "week_start": 1,
  "timezone": "Asia/Bangkok"
}
```

Only `title` and `bg_color` are currently rendered. The other values are
reserved in schema version 1 so web, mobile, and firmware agree when calendar
behavior is expanded.

## 5. Compiled device document

The firmware should not understand database tables or canonical nested forms.
The API compiles them into the existing device-compatible document.

Example:

```json
{
  "pages": ["clock", "crypto", "slideshow", "weather", "calendar", "profile"],
  "display_mode": "static",
  "page_delay_s": 10,
  "brightness": 80,
  "clock": {
    "format_24h": true,
    "show_seconds": true,
    "time_color": "#00D1FF",
    "sec_color": "#FF9500",
    "date_color": "#848E9C",
    "bg_color": "#0B0E11"
  },
  "profile": {
    "name": "Natthapong Suwanjit",
    "nickname": "SAT",
    "role": "Developer",
    "name_color": "#F0B90B",
    "avatar": "pages/profile/assets/avatar.png"
  }
}
```

The complete boot response is an envelope:

```json
{
  "revision": 42,
  "sha256": "<compiled-json-sha256>",
  "config": {},
  "assets": [
    {
      "id": "<asset-version-id>",
      "page": "profile",
      "key": "avatar",
      "local_path": "pages/profile/assets/avatar.png",
      "download_url": "<short-lived-signed-url>",
      "sha256": "<file-sha256>",
      "size_bytes": 23104,
      "content_type": "image/png"
    }
  ]
}
```

The server signs URLs only when generating this response. The database contains
the private object path, not `download_url`.

## 6. Revision and transaction rules

Every write uses one database transaction:

1. Lock `DeviceConfigHead` for the target device.
2. Check the client's `baseRevision`.
3. Validate the page config against `PageSettingSchema`.
4. Update the touched page/system/asset row.
5. Increment the page revision when applicable.
6. Increment the global head revision exactly once.
7. Rebuild canonical `compiledConfig` and SHA-256.
8. Insert `DeviceConfigRevision`.
9. Set `DeviceSyncState.desiredRevision` and status to `pending`.
10. Commit.
11. Publish MQTT notification after commit.

Concurrent write contract:

```text
Client sends baseRevision=41
Server current revision=42
→ HTTP 409 CONFIG_REVISION_CONFLICT
```

The response includes the current revision and current page document. The
client asks the user to reload or deliberately overwrite. Silent last-write-wins
must not be used.

## 7. Device synchronization

### Boot

1. Device loads local last-known-good revision.
2. Device authenticates with its device token.
3. Device calls:

```text
GET /api/v1/device/bootstrap?revision=<local-revision>
```

4. If unchanged, API returns `204 No Content`.
5. If newer, API returns compiled config plus the asset manifest.
6. Device downloads only missing/hash-mismatched assets.
7. Device atomically writes config and assets to staging.
8. Device activates the new revision.
9. Device publishes/calls an ACK containing revision and SHA-256.

### Web/Mobile save

1. Client reads page settings from OCI API.
2. Client saves one page using its current `baseRevision`.
3. API commits Supabase transaction.
4. API publishes a small MQTT message:

```json
{
  "type": "config_changed",
  "revision": 43
}
```

5. Online device fetches revision 43 over HTTPS and acknowledges it.
6. Offline device obtains revision 43 at the next boot/reconnect.

MQTT must not carry PNG/GIF data or the full long-term configuration document.

## 8. API contract

User/admin endpoints:

```text
GET  /devices/:deviceId/config
GET  /devices/:deviceId/pages/:pageSlug
PUT  /devices/:deviceId/pages/:pageSlug
PUT  /devices/:deviceId/system
PUT  /devices/:deviceId/pages/order

GET    /devices/:deviceId/pages/:pageSlug/assets
POST   /devices/:deviceId/pages/:pageSlug/assets/:assetKey
DELETE /devices/:deviceId/pages/:pageSlug/assets/:assetId

GET  /devices/:deviceId/sync-state
GET  /devices/:deviceId/revisions
POST /devices/:deviceId/revisions/:revision/restore
```

Example page write:

```json
{
  "baseRevision": 42,
  "schemaVersion": 1,
  "config": {
    "nickname": "SAT",
    "role": "Developer"
  }
}
```

Device-only endpoints:

```text
GET  /device/bootstrap?revision=42
POST /device/config-ack
POST /device/config-error
```

Device endpoints authenticate with the minted device token, never a user JWT.

## 9. Storage and security

- Bucket `device-assets` is private.
- Browser upload goes through OCI API or a narrowly scoped signed-upload URL.
- API verifies ownership before generating upload/download signatures.
- API re-encodes Profile avatars to PNG 132×132.
- API verifies MIME type from file bytes, not filename alone.
- API sets page-specific size/dimension limits.
- Device verifies SHA-256 before activation.
- Never store Supabase service-role keys, device tokens, signed URLs, or claim
  secrets in page config.
- Web and Mobile use the OCI API as the authorization boundary.
- Add `User.authSubject` so ownership can be tied to Supabase JWT `sub`, not
  mutable email.

RLS remains enabled as defense in depth. Ordinary `anon`/`authenticated`
clients receive no direct table-write policies while the API-only model is in
use.

## 10. Migration from current `Device.settings`

1. Add the new tables without deleting existing columns.
2. Backfill one `DeviceConfigHead` per device:
   - `revision = Device.settingsVersion`;
   - system keys from top-level `Device.settings`;
   - one `DevicePageSettings` row for each page object;
   - `enabled/position` from the current `pages` array.
3. Treat `Device.settings` as the generated compatibility snapshot.
4. During transition, API writes new tables and regenerates
   `Device.settings/settingsVersion` in the same transaction.
5. Change Web User/Mobile/Admin reads from MQTT/local config to REST API.
6. Import existing device-only Profile data once.
7. Move avatar/GIF/photo files into `device-assets`, create metadata rows, and
   produce the first asset manifest.
8. After every supported client and firmware uses the new API, remove direct
   writes to legacy `Device.settings`.

Important current-device limitation: firmware `4321d17-dirty` accepts settings
but does not expose a `get_settings` command. Existing Profile values that are
only on the physical clock cannot be recovered by Supabase automatically. A
one-time firmware read-config command or same-LAN export is required before
declaring migration complete.

## 11. Recommended implementation order

1. Add `authSubject` and the seven configuration/asset/sync tables.
2. Implement schema validation and config compiler.
3. Backfill current Supabase data and dual-write the legacy snapshot.
4. Switch Web User settings GET/PUT to OCI REST.
5. Add asset upload, private storage, manifest generation, and web preview.
6. Add device bootstrap/ACK protocol.
7. Add one-time import of settings currently stored only on the clock.
8. Switch Mobile and Admin to the same REST contracts.
9. Remove MQTT `device.settings.get` and use MQTT only for change notification.
